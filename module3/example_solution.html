    Модуль 3. Проектирование, разработка и оптимизация веб-приложений - Пошаговое выполнение задания

    1 Доработка базы данных под задачи заказчика

    1.1. Исправить слабые места текущей схемы (минимальный обязательный апгрейд)

    В вашей БД сейчас есть 3 важных проблемы:

    1. users.login допускает NULL (это плохо для авторизации).
    2. users.email не уникален (можно зарегистрировать один email несколько раз).
    3. reviews.created_at — int, и в данных стоит 0 (нельзя нормально сортировать/показывать дату).

    1.2. Создать файл миграции

    1. Создать файл: db/migrations/001_fix_schema.sql
    2. Вставить в него:
    USE korochki;

-- 1) login сделать обязательным
ALTER TABLE users
  MODIFY login VARCHAR(50) NOT NULL;

-- 2) email сделать уникальным
ALTER TABLE users
  ADD UNIQUE KEY uq_users_email (email);

-- 3) reviews.created_at сделать timestamp с автозаполнением
ALTER TABLE reviews
  MODIFY created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP;

-- 4) reviews.text расширить (200 символов часто мало)
ALTER TABLE reviews
  MODIFY text TEXT NOT NULL;

-- 5) payment_methods.name сделать уникальным (как справочник)
ALTER TABLE payment_methods
  ADD UNIQUE KEY uq_payment_methods_name (name);

    1.3.Проверка результата

    • В phpMyAdmin → вкладка Structure:

    • users.login = NOT NULL

    • users.email имеет UNIQUE
    • reviews.created_at = TIMESTAMP DEFAULT CURRENT_TIMESTAMP

    1.4. Закрепить «курсы только из списка» на уровне БД (правильно по требованиям)

    Сейчас applications.course_name — свободный текст. Это позволяет писать любые курсы, что противоречит заданию (там 3 фиксированных курса).

    1.4.1. Ввести справочник курсов и привязать заявки к нему

    1. Создать файл: db/migrations/002_courses_dictionary.sql

    2. Вставить:

    USE korochki;

-- 1) создать справочник курсов
CREATE TABLE IF NOT EXISTS courses (
  id INT NOT NULL AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(200) NOT NULL UNIQUE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 2) заполнить 3 курса из задания
INSERT IGNORE INTO courses (name) VALUES
('Основы алгоритмизации и программирования'),
('Основы веб-дизайна'),
('Основы проектирования баз данных');

-- 3) добавить course_id в applications
ALTER TABLE applications
  ADD COLUMN course_id INT NULL AFTER user_id;

-- 4) проставить course_id по старому course_name
UPDATE applications a
JOIN courses c ON c.name = a.course_name
SET a.course_id = c.id;

-- 5) сделать course_id обязательным и добавить FK
ALTER TABLE applications
  MODIFY course_id INT NOT NULL,
  ADD CONSTRAINT fk_applications_course
    FOREIGN KEY (course_id) REFERENCES courses(id)
    ON UPDATE CASCADE;

-- 6) убрать старое поле (после успешного обновления)
ALTER TABLE applications
  DROP COLUMN course_name;

    1.4.2. Проверка результата

    • В applications больше нет course_name
    • Есть course_id и таблица courses с 3 записями
    • Нельзя вставить заявку с курсом “левым текстом”

    1.5. Закрепить правило «отзыв только после завершения обучения»

    Правильно реализовывать это так:

    • отзыв разрешён, если хотя бы одна заявка пользователя имеет статус “Обучение завершено”.

    У вас статусы уже есть (application_statuses), это хорошо. Дорабатывать БД дополнительно не обязательно.

    2. Технологичность и масштабируемость приложения

    2.1. Привести API к «единым правилам»

    2.1.1. Убрать дубли и сделать статику единообразной

    В вашем app.py есть два роута для /styles.css — оставить один.

    1. Удалить дублирующий обработчик /styles.css (оставить только один).
    2. Удалить print(hashlib.sha256(...)) из рабочего сервера (это мусор в проде).

    2.1.2. Использовать относительные fetch

    Во фронтенде использовать:

    • fetch("/api/login"), а не http://127.0.0.1:5000/api/login

    Это делает проект переносимым на другой хост/порт.

    2.2. Подстроить backend под новый courses справочник

    После миграции A2 поменяется запрос выдачи заявок и создание заявки.

    2.2.1. Исправить SELECT в /api/applications/my

    Заменить выборку курса:

    • было: a.course_name
    • станет: c.name AS course_name + JOIN

    Пример:

    SELECT
  a.id,
  c.name AS course_name,
  DATE_FORMAT(a.start_date, '%Y-%m-%d') AS start_date,
  pm.name AS payment_method,
  st.name AS status,
  DATE_FORMAT(a.created_at, '%Y-%m-%d %H:%i:%s') AS created_at
FROM applications a
JOIN courses c ON c.id = a.course_id
...

    2.2.2. Исправить /api/applications (создание заявки)

    Вместо сохранения course_name сохранять course_id:

    • по входному course_name найти id:

    • SELECT id FROM courses WHERE name=%s

    • потом вставлять course_id.

    3. Адаптивный дизайн

    Вы уже начали mobile-first. Чтобы на всех страницах не было “разъезда”:

    3.1. Привести разметку страниц к единой структуре

    3.1.1. Описание

    На каждой странице придерживаться:

    • <body class="container">
    • заголовок h1
    • основной блок в .card
    • сообщения — #msg с классом .msg
    • формы: .field + label + input/select/textarea
    • кнопки в .actions

    Это решает 90% ваших ошибок вида textContent of null.

    4. Высокое качество кода (устранение типовых JS-ошибок)

    Вы уже столкнулись с системной проблемой: на разных страницах нет #msg, но код его ожидает.

    4.1. Ввести стандарт функции сообщений на всех страницах

    4.1.1. Описание

    На каждой странице иметь:

    <div id="msg" class="msg"></div>

    И в JS:

    function setMsg(text, type="") {
  const el = document.getElementById("msg");
  if (!el) return;
  el.className = type ? `msg ${type}` : "msg";
  el.textContent = text || "";
}

    После этого ошибки “Cannot set properties of null” исчезают по всему проекту.

    Блок 5. Анимации и микроанимации (UX)

    5.1. Добавить микроанимации без библиотек (достаточно для демо)

    5.1.1. Добавить в styles.css

    /* microanimations */
button, a, input, select, textarea {
  transition: transform 120ms ease, filter 120ms ease, box-shadow 120ms ease, border-color 120ms ease;
}

button:active {
  transform: scale(0.98);
}

.card {
  transition: transform 160ms ease, box-shadow 160ms ease;
}

.card:hover {
  box-shadow: 0 10px 26px rgba(0,0,0,0.08);
}

input:focus, select:focus, textarea:focus {
  box-shadow: 0 0 0 4px rgba(26, 115, 232, 0.12);
}

    5.1.2. Проверка результата

    • кнопки “живые”
    • карточки слегка реагируют на наведение
    • фокус заметный и аккуратный
